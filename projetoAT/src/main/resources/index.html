Após muito pensar se daria continuidade ou não ao projeto realizado até o TP 3, decidi pedir ajuda à minha namorada e ela me deu a sugestão de fazer um sistema de venda de cosméticos. A ideia dela foi excelente (obrigado por ela meu amor <3), pois bateu com o requisito dos números de classe do projeto e a oportunidade de utilizar algumas regras de negócio também.
Para implementar o projeto, foram elaboradas as classes de Produto, Endereco, Cliente, Pedido e Maquiagem e Skincare (subclasses de produto). Embora não se pareça nada com um sistema utilizado no cotidiano, a ideia é apenas manipular os dados do sistema, fazendo alguns cálculos básicos, relacionando as classes e repassando os dados obtidos para uma estrutura de armazenamento ( a escolhida foi o Map para diminuir a complexidade do projeto).
Com isso, o projeto foi separado entre 4 camadas:

  • Camada de Domínio - Utilizada para estabelecer o modelo padrão e regras das nossas classes de negócio.
  • Camada de Serviço - Faz a conexão entre as camadas e a estrutura de armazenamento.
  • Camada de Controle - É a responsável por fazer a interação com o usuário.
  • Camada de Client - Sua função é estabelecer conexão com uma API externa e consumi-la.

As classes de domímio foram as classes cruciais para um sistema de vendas: Produto (e suas subclasses Maquiagem e Skincare), Cliente, Endereco e Pedido. Cada classe possui um método de retorno toString e as classes Produto e Pedido possuem métodos para calcularem o desconto de um produto, baseado na regra de negócio estabelecida. Para tanto, caso o valor unitário do produto seja maior que R$ 250,00 reais, o produto ganha 10% de desconto e, caso o valor final do pedido excede mil reais, 20% de desconto é aplicado ao produto final.

Dentro das classes de serviço foram incluídos métodos genéricos para realizar um CRUD (exceto pelo Update) das classes, que contam com os métodos de inclusão, exclusão, listar todos os elementos contidos na estrutura de dados, e listar somente um elemento, trazido através de seu ID. A partir dessa classe, são estabelecidas as rotas, através da camada de controle. Essa camada será a responsável pela interação entre servidor e usuário, onde o usuário solicita uma operação e o servidor retorna uma resposta.

Por fim, a classe Client do sistema faz o consumo da API externa do ViaCep, onde, através dessa API, um método chamado "obterPorCep" é criado, puxando automaticamente os atributos que a classe Endereco precisa para ocorrer a entrega corretamente.

Os testes das funcionalidades do sistema podem ser feitos através do arquivo localizado aqui. Todos foram realizados no Postman, permitindo assim chegar às funcionalidades da camada do back-end, antes de ser implementada pelo time de front-end.
